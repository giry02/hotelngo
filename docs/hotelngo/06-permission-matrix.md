# HotelNGo 인증·권한·데이터 범위 명세

## 1. 인증 영역

| 영역 | 주체 | 인증 방식 | 기본 범위 |
|---|---|---|---|
| B2C | 익명/회원 | 익명 세션, 회원 OIDC/OAuth 기반 세션 | 본인 프로필·일정·주문·예약 |
| 파트너 | 파트너 구성원 | MFA 권장/고위험 작업 필수, 조직 선택 | 소속 `partnerId` 및 허용 Provider/Place |
| 플랫폼 관리자 | 운영 구성원 | SSO + MFA | 역할별 플랫폼 기능 |
| Hotel_PMS Guest/회원 | 호텔별 PMS 고객 | PMS 정책 | 해당 PMS tenant 내부; HotelnGo 로그인 불가 |
| 시스템 연동 | PMS/외부 공급자 | mTLS 또는 회전 가능한 서비스 토큰 | 지정 `supplierId/tenantId` |
| AI 실행기 | 내부 워크로드 | 워크로드 ID | 허용 도구와 읽기 범위 |

HotelnGo B2C, 파트너, 플랫폼 관리자, Hotel_PMS는 인증·세션·쿠키·라우팅 영역을 분리한다. 한 영역의 토큰이나 비밀번호를 다른 영역에서 수용하지 않는다. `PmsGuestLink`는 업무 레코드 참조일 뿐 인증 연동이 아니다.

## 2. 역할 정의

### 2.1 B2C

- `ANONYMOUS`: 공개 조회, 익명 장바구니
- `CUSTOMER`: 본인 일정·찜·장바구니·주문·예약·리뷰·문의

### 2.2 파트너

- `PARTNER_OWNER`: 조직/지급계좌/역할을 포함한 전체 관리
- `PARTNER_ADMIN`: 구성원, 사업장, 상품, 판매, 예약 운영
- `PROPERTY_MANAGER`: 지정 Place의 공개정보·상품·가격·재고·예약
- `RESERVATION_STAFF`: 지정 Place 예약/문의 처리
- `CONTENT_EDITOR`: 공개정보·미디어·콘텐츠 변경 요청
- `FINANCE_VIEWER`: 매출·정산·지급 내역 읽기
- `PARTNER_VIEWER`: 허용 범위 읽기 전용

### 2.3 플랫폼 관리자

- `PLATFORM_SUPER_ADMIN`: 역할/시스템 정책을 포함한 전체 관리; 일상 운영에 사용 금지
- `CATALOG_ADMIN`: Provider/Place/중복/병합/수집
- `PARTNER_REVIEWER`: 입점·클레임·변경 요청 심사
- `CONTENT_ADMIN`: 스토리·랜드마크·미디어·번역·발행
- `COMMERCE_OPERATOR`: 주문·예약·변경·취소·대사
- `FINANCE_ADMIN`: 결제·환불·정산·지급
- `CUSTOMER_SUPPORT`: 고객/예약 조회와 제한된 지원 작업
- `AI_OPERATOR`: RAG 원천·AI 실행·평가·차단
- `AUDITOR`: 감사자료 읽기 전용

## 5. 데이터 범위와 소유권 규칙

1. 모든 파트너 데이터 행은 `partnerId` 또는 그에 귀속되는 불변 참조를 갖는다.
2. 사용자 입력의 `partnerId/providerId/placeId`를 신뢰하지 않고 인증 컨텍스트와 서버 조회 결과로 범위를 결정한다.
3. `Provider` 소유권과 `Place` 운영권은 별도로 부여할 수 있다. 한 사용자가 Provider를 볼 수 있어도 모든 Place를 변경할 수 있는 것은 아니다.
4. `INTERNAL_PMS` 호텔은 PMS 매핑이 원천인 판매 필드를 파트너가 직접 변경할 수 없다.
5. `PLATFORM_CURATED` 필드는 플랫폼 검수 전까지 공개 원본을 변경하지 않는다.
6. 합병된 Place의 이전 ID는 리다이렉트/감사 참조로 보존하고 다른 파트너가 재사용할 수 없다.
7. 검색 인덱스, 캐시, 벡터 저장소도 원본과 같은 공개상태·파트너 범위·삭제 정책을 반영한다.

## 7. 고위험 작업 통제

다음 작업은 권한 확인에 더해 사유, 이전/이후 값, 행위자, 시각, trace ID를 감사로그에 기록한다.

- Provider/Place 병합·분리, 파트너 정지, 소유권 클레임 승인
- 공개 콘텐츠 발행/회수, AI 생성 콘텐츠 자동 발행 정책 변경
- 예약 강제 상태 변경, 공급 예약 수동 연결/해제
- 결제 취소, 환불, 정산 확정/보류/조정, 지급계좌 변경
- 구성원 권한 승격, 서비스 자격증명 생성/회전/폐기
- 개인정보 조회/내보내기/삭제 처리

고위험 작업은 `SUPER_ADMIN` 보유만으로 자동 허용하지 않고 업무 목적의 세부 capability도 검사한다.

## 8. AI 권한 경계

- AI 검색은 공개 또는 사용자가 접근 가능한 데이터만 반환한다.
- 내부 메모, 정산, 직원, Folio, 다른 고객의 예약/여정은 검색·프롬프트 컨텍스트에서 제외한다.
- 실시간 도구는 사용자 세션과 동일한 범위로 실행한다.
- 예약·결제·취소·환불은 AI가 초안/설명만 제공하고 명시적 사용자 확인 후 일반 API가 수행한다.
- AI 생성 스토리의 발행은 `CONTENT_ADMIN` 승인 또는 사전 정의된 저위험 자동발행 정책이 필요하다.
- 모든 도구 호출은 입력 요약, 결과 참조, 모델/프롬프트 버전, 최종 사용자 영향을 추적한다.

## 9. 권한 테스트 기준

- 역할별 허용/거부 양방향 테스트를 작성한다.
- 다른 `partnerId`, `providerId`, `placeId`, `tenantId`를 요청 경로·본문·쿼리에 주입해도 데이터가 노출되지 않아야 한다.
- 목록 합계, 검색 인덱스, 자동완성, 파일 URL, 로그에서도 범위 누출이 없어야 한다.
- 역할 변경/탈퇴/파트너 정지 시 기존 세션과 API 키가 정책 시간 내 무효화되어야 한다.
- 감사로그는 일반 운영자가 수정·삭제할 수 없어야 한다.
