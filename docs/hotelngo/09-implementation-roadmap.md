# HotelNGo 단계별 구현·전환 계획

## 1. 추진 원칙

- 기존 Hotel_PMS를 재작성하거나 복제하지 않고 공급 원천으로 연결한다.
- 공개 카탈로그, 공급, 예약, 결제, 콘텐츠, AI의 책임을 분리한다.
- 각 릴리스는 화면만이 아니라 계약·데이터·권한·테스트·운영 관측까지 완료해야 한다.
- 실제 서버/DB가 없는 기능은 Mock을 실연동 완료로 표시하지 않는다.
- 단계별 기능 플래그, 마이그레이션, 롤백, 대사 절차를 준비한다.
- 코드 변경은 작은 단위로 수행하고 관련 문서·계약·테스트를 함께 변경한다.

## 2. 기준 기술 방향

### Release 1

현재 선택을 반영해 의존성을 최소화한 정적 웹 기반으로 시작한다.

- HTML/CSS/ES modules
- 브라우저 라우팅 또는 단순 페이지 라우팅
- 독립 JSON fixture + `HotelSupplyAdapter` Mock
- 화면과 어댑터의 경계를 서비스 모듈로 분리
- Node 기반 테스트 러너/정적 검사 도입 가능

정적 구현은 빠른 수직 슬라이스 검증용이다. 파트너·관리자·예약·결제의 실제 다중 사용자 기능을 `localStorage`로 확장하지 않는다.

### Release 2 이후

- B2C 웹: SSR/SEO와 다국어를 지원하는 웹 프레임워크
- 플랫폼 API: Node.js/TypeScript 모듈형 API
- 데이터: PostgreSQL, 마이그레이션, 감사/Outbox
- 캐시/작업: Redis 계열 캐시, 지속성 있는 작업 큐
- 검색: 키워드/지리/필터 검색엔진
- 객체 저장소/CDN: 미디어와 업로드
- 관측: 구조화 로그, 메트릭, 추적, 오류 수집

구체 제품은 Gate 0 ADR에서 결정하며, 제품명보다 계약·데이터 소유권·교체 가능성을 우선한다.

## 4. Gate 0 — 분석·설계·프로젝트 기반

### 작업

- 현재 Hotel_PMS 코드/fixture/API/localStorage/테스트 기준선 기록
- 이 문서 세트와 요구사항 ID 승인
- 다음 ADR 작성: 저장소/모듈 구조, 런타임, DB/검색, 인증 영역, 공급 어댑터, 예약/결제 오케스트레이션, 다국어/시간대
- 환경(dev/test/staging/prod), 비밀정보, 로깅, 마이그레이션/백업 원칙 정의
- Hotel_PMS의 기존 변경사항과 미추적 테스트 산출물 보호

### 완료 게이트

- 모든 원문 장이 요구사항 ID와 후속 문서에 매핑됨
- 코드로 확인된 사실과 백엔드 생성 시 확정할 가정이 구분됨
- Release 1의 fixture 스키마·화면·계약 테스트가 승인됨
- PMS 회귀 테스트와 변경 금지 범위가 기록됨

## 5. Release 1 — 공통 기반과 PMS Mock 조회

원문 33장의 첫 구현 요청과 현재의 “PMS 방식 유지 + 조회 수직 슬라이스” 결정을 구현한다.

### 작업 묶음

1. 앱 셸과 공통 UI
   - 반응형 B2C 헤더/탐색/오류·로딩 컴포넌트
   - 다국어 키, 통화/날짜 표시 기반
2. 공급 계약
   - 공급 방식 enum과 `HotelSupplyAdapter`
   - canonical 호텔/객실상품/Offer/가용성 모델
   - Mock 어댑터와 오류/신뢰 상태
3. 독립 fixture
   - 정상, 매진, 부분 누락, 오래된 데이터, 공급 미응답 시나리오
   - 물리 호실·직원·Folio 제외
4. B2C 수직 슬라이스
   - `GET /api/v1/public/hotels`에 대응하는 검색 서비스/화면
   - 호텔 상세와 객실유형별 요금/재고
   - 예약 API 미연결 상태와 조회 전용 CTA
5. 품질
   - 어댑터 계약·매핑·테넌트 격리·오류 상태 테스트
   - 접근성/반응형/브라우저 회귀

### 완료 게이트

- 검색 → 상세 → 객실/요금 조회가 fixture가 아닌 어댑터를 통해 흐름
- 전체 숙박일 가격/재고가 없을 때 판매가능 표시 0건
- PMS 내부 식별자 노출 0건, Hotel_PMS 데이터 변경 0건
- Mock 쓰기 메서드가 명시적 `UNSUPPORTED_OPERATION` 반환

## 6. Release 2 — 실 플랫폼 기반·파트너·관리자

### 작업 묶음

- 실 API/DB, 계정, 파트너/관리자 인증 영역, RBAC/범위 검사
- Provider/Place/Product/Offer/Availability의 기본 CRUD와 상태 전이
- 파트너 가입·사업자/정산 문서·심사
- 플랫폼 선등록, 가져오기/오류 보고, 중복 후보/병합
- 소유권 클레임, 변경 요청, 최신성 검증/재확인
- 파트너 호텔/업종별 필드의 정보성 상품 관리
- 스토리/랜드마크 편집·검수·발행의 최소 관리자 기능
- 감사로그, Outbox, 작업 큐, 미디어 업로드, 검색 색인 동기화

### 데이터 이행

- Release 1 fixture의 공개 샘플을 seed 데이터로 변환하되 운영 PMS 원천으로 오인하지 않도록 표시
- 식별자와 slug 리다이렉트 규칙 고정
- 검색 인덱스는 DB에서 재생성 가능하게 구성

### 완료 게이트

- 서로 다른 파트너 데이터 격리 테스트 통과
- 심사/클레임/변경요청/병합에 감사이력과 허용 상태 전이 존재
- 실패한 비동기 작업을 재시도/재생할 수 있음
- 관리자 없이 파트너가 플랫폼 큐레이션 필드를 즉시 공개하지 못함

## 7. Release 3 — 실제 호텔 공급과 예약

### 작업 묶음

- Hotel_PMS 실제 서버/DB와 `/api/v1/channel/*` 읽기 API
- 서비스 인증, tenant 범위, 추적/호출제한, 계약 버전
- Shadow 비교와 기능 플래그 전환
- 독립호텔 `PARTNER_MANAGED` 객실상품·요금·재고
- 양 공급자에 공통 hold/release/create/modify/cancel/status/reconcile 계약
- HotelNGo Booking과 PMS Reservation을 `PmsReservationLink`로 분리
- 주문/결제/예약 오케스트레이션, 멱등, 타임아웃, 대사, 수동 복구
- 고객 예약 조회, 바우처, 취소/변경 진입점

### 완료 게이트

- 실제 채널 API가 `07-pms-integration.md`의 shadow 기준 통과
- 동시 hold/예약에서 초과판매 0건
- 공급자 타임아웃을 확정으로 표시하지 않음
- 결제/예약 부분 실패가 대사 및 수동 복구 가능
- Hotel_PMS 운영·예약 회귀 게이트 통과

## 8. Release 4 — 공개 여행 탐색 MVP

원문 28장의 ‘반드시 포함’ 범위를 이 단계까지 완성해 첫 공개 MVP로 삼는다.

### 작업 묶음

- B2C 홈의 목적지 검색, 추천 여행지, 카테고리, 스토리
- 랜드마크 목록/지도/상세와 관련 상품·스토리 연결
- 몰입형 스토리 블록, 저장/공유/일정담기
- 직접 일정 만들기, 날짜별 타임라인, 순서 변경, 이동/운영시간/중복 검증
- 호텔 외 업종(골프, 렌터카/기사차량, 음식점/카페, 마사지/스파, 투어/체험, 공연/티켓, 교통)의 정보성 페이지와 파트너 기본 관리
- 찜, 내 일정, 내 예약의 기본 회원 기능

### 완료 게이트

- 스토리/랜드마크/상품을 실제 엔티티 참조로 일정에 추가 가능
- 비공개/회수/오래된 콘텐츠가 공개 검색에서 정책대로 처리됨
- 정보성 업종이 HotelNGo 내부 예약 가능 상품으로 오인되지 않음
- 시간대·운영시간·이동시간·중복 일정 검증 통과

## 9. Release 5 — 멀티업종 거래

### 작업 묶음

- 업종별 Inventory/Offer/Booking 어댑터
- 투어 회차, 공연 좌석/회차, 골프 티타임, 렌터카 차량/보험, 교통 운행편, 음식점 슬롯 등 업종별 재고 모델
- 통합 장바구니와 가격 스냅샷/만료/재검증
- Order와 항목별 Booking 분리, 부분확정·부분실패·부분취소·부분환불
- 공급자별 취소/변경, 알림/바우처, 파트너 운영화면
- 지급·수수료·정산 대사

### 완료 게이트

- 한 주문의 여러 공급자 결과를 개별 예약으로 추적 가능
- 실패 항목 때문에 성공 항목 상태가 손상되지 않음
- 멱등 재호출과 이벤트 중복 처리에서 중복 예약/결제 없음
- 부분 환불/정산이 주문·예약·결제 원장과 대사됨

## 10. Release 6 — RAG

### 작업 묶음

- 승인 원천 수집, 구조 청킹, 임베딩/키워드 색인, 버전/회수
- 지역·언어·카테고리·공개상태·유효기간 필터의 하이브리드 검색
- 출처/근거 표시와 검색 관리자 화면
- 오프라인 평가 세트, 지표 대시보드, 비용/지연 관측
- 삭제/회수/권한 변경의 인덱스 반영 테스트

### 완료 게이트

- 비공개/삭제/다른 사용자 데이터 검색 노출 0건
- 가격·재고·예약 상태가 RAG에서 확정값으로 사용되지 않음
- 검색 품질 임계치와 근거 정확성 기준 통과

## 11. Release 7 — AI 여행 플래너·스토리·개인화

- 일정 제약 검증, 대체안, 근거/최신 확인시각 표시
- AI 스토리 블록 초안, 실제 엔티티 연결, 편집자 승인
- 개인화 추천, 피드백/신고, 프롬프트·모델·도구 버전 관리

### 완료 게이트

- AI의 직접 예약/결제/취소 0건
- 일정 제약과 실시간 가격/재고 재검증 기준 통과
- 프롬프트 인젝션/범위 누출/삭제 원천 회귀 통과
- 기능 플래그로 AI를 끄고 일반 검색/일정으로 복귀 가능

## 13. 릴리스 공통 Definition of Done

- 관련 요구사항 ID, API/OpenAPI, 데이터 모델, 화면, 권한 문서가 일치한다.
- 단위·계약·통합·E2E·보안·접근성 테스트가 위험에 비례해 존재하고 CI를 통과한다.
- 마이그레이션, seed/fixture, 롤백 또는 안전한 기능 비활성화 절차가 있다.
- 로그/메트릭/추적/경보와 운영 런북이 있다.
- 개인정보·비밀정보·권한·테넌트 범위 검토를 통과한다.
- 사용자가 보는 오류/빈 상태/오래된 데이터/부분 실패가 설계되어 있다.
- 임시 Mock/가정은 코드와 문서에 명시되고 실구현 전환 조건이 추적된다.

## 14. 즉시 구현 백로그 순서

1. 저장소 구조와 품질 도구 구성
2. canonical 공급 타입/DTO/오류 타입
3. 정상·매진·오래된·실패 fixture
4. Mock `HotelSupplyAdapter`
