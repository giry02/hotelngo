# HotelNGo 테스트·인수 전략

## 1. 목표

테스트는 화면이 보이는지만 확인하지 않고 데이터 원천, 권한 범위, 가격/재고 진실성, 예약·결제의 멱등성, 부분 실패 복구, 콘텐츠/AI의 근거와 회수를 검증한다. 각 테스트 결과는 요구사항 ID와 릴리스 게이트에 연결한다.

## 2. 테스트 계층

| 계층 | 대상 | 실행 시점 |
|---|---|---|
| 정적 검사 | 형식, 타입, 린트, 금지 의존성/비밀정보 | 모든 변경 |
| 단위 | 도메인 규칙·상태 전이·변환·권한 | 모든 변경 |
| 계약 | 공급/결제/지도/검색/메시징/AI 도구 | 계약 변경 및 CI |
| 통합 | API+DB+큐+캐시+어댑터 | PR/통합 환경 |
| E2E | 주요 사용자/운영 흐름 | PR 핵심경로, staging 전체 |
| 회귀 | Hotel_PMS 및 이미 출시된 흐름 | 연동/릴리스 전 |
| 비기능 | 성능·보안·접근성·복구·AI 평가 | 릴리스 게이트 |

## 3. 단위 테스트

### 3.1 공급·호텔

- 공급 방식별 어댑터 선택과 미지원 동작
- Tenant → 공개 Hotel, RoomType → RoomProduct, 요금 → Offer 변환
- 요청한 모든 숙박일의 요금/재고 교집합
- 점검·사용중·판매중지·중복 기간 객실 제외
- 통화·세금 포함 여부·취소정책·총액 계산과 알 수 없음 처리
- 물리 호실/직원/Folio 제거
- FRESH/STALE/UNKNOWN 판정과 판매 차단

### 3.2 카탈로그·파트너

- Provider/Place/Product/Offer 책임과 불변조건
- 상태 전이: 초안·심사·승인·반려·중지·회수
- 중복 후보 점수, 병합 후 리다이렉트/참조 보존
- 소유권 클레임·변경요청의 허용 전이와 필드 소유권
- 최신성 만료·재확인·비공개 처리
- 업종별 필수 필드/정책 검증

### 3.3 커머스

- Cart 가격 스냅샷 만료와 재검증
- Order/Booking/Payment/Refund/Settlement의 분리와 매핑
- 홀드 만료, 멱등성 키 재호출/충돌
- 부분확정·부분실패·부분취소·부분환불 집계
- 예약/결제 보상 및 대사 상태 전이

### 3.4 콘텐츠·일정·AI

- 스토리 블록 순서/타입/실제 엔티티 참조
- 장소 운영시간·시간대·이동시간·일정 중복/예산 검증
- RAG 공개상태/언어/지역/유효기간 필터
- AI 도구 JSON Schema와 권한, 실시간 재검증 필수 규칙
- AI 생성 콘텐츠 기본 `DRAFT`와 근거/버전 기록

## 4. 계약 테스트

### 4.1 `HotelSupplyAdapter`

모든 공급자는 동일 fixture suite를 통과한다.

- 검색·상세·가용성·요금의 필수 필드와 의미
- 날짜 경계, 1박/다박, 성인/아동/복수 객실
- 매진, 일부 날짜 누락, 가격 변경, 오래된 데이터, timeout
- 통화/세금/취소 규칙과 `verifiedAt`
- 식별자 안정성 및 내부 ID 비노출
- 쓰기 지원 공급자의 hold/create/modify/cancel/status/reconcile

### 4.2 외부 계약

- Hotel_PMS `/api/v1/channel/*`: 인증, tenant, 응답 버전, 오류/timeout, 멱등
- 결제: 승인/실패/timeout/webhook 중복/취소/부분환불
- 지도/경로: 좌표·시간대·호출한도·실패 대체
- 메시징: 이메일/SMS/푸시 중복 방지와 실패 재처리
- 검색/벡터: 공개상태·삭제·권한 필터 보존

소비자 주도 계약은 버전별로 저장하며 호환성 파괴 변경을 CI에서 차단한다.

## 5. 통합 테스트

- B2C 검색 → 어댑터 → 캐시 → 공개 응답의 원천/최신성 전파
- 공개 호텔 상세 → 객실 Offer → 재검증의 가격/재고 일관성
- 파트너 가입 → 문서 제출 → 관리자 승인 → 로그인/사업장 접근
- 플랫폼 선등록 → 소유권 클레임 → 승인 → 필드 소유권 전환
- CSV/API 수집 → 검증 오류 → 수정 재처리 → 중복 병합 → 검색 반영
- 독립호텔 상품/가격/재고 → B2C 조회 → hold → 예약 → 파트너 예약함
- PMS 예약 생성 timeout → `PENDING_SUPPLIER` → 상태조회/대사 → 확정 또는 복구
- 다중 공급자 장바구니 → 결제 → 일부 확정/실패 → 부분환불/정산
- 스토리 → 장소/상품 → 일정 추가 → 검증 → 장바구니 후보
- 원천 회수/삭제 → 검색/캐시/RAG/공개 페이지 비활성화

## 6. E2E 시나리오

### Release 1

1. 목적지/날짜/인원으로 호텔 검색
2. 호텔 상세에서 공개정보와 정책 확인
3. 객실유형별 총액·취소조건·재고·확인시각 확인
4. 매진, 공급 미응답, 오래된 캐시 화면 구분
5. 예약 CTA가 실제 예약처럼 동작하지 않음을 확인

### Release 2~3

1. 파트너 가입·심사·사업장 등록·상품 제출
2. 플랫폼 선등록 장소의 소유권 클레임과 운영권 인수
3. 관리자의 중복 장소 병합과 기존 URL/참조 보존
4. PMS 호텔 예약·변경·취소·대사
5. 독립호텔 예약과 파트너 예약 처리
6. 다른 파트너/테넌트의 URL·ID를 주입한 접근 차단

### Release 4~7

1. 스토리 탐색 → 랜드마크 상세 → 일정 추가
2. 직접 일정의 이동/운영시간/중복 충돌 수정
3. 여러 업종 장바구니의 부분확정·부분실패·부분취소
4. AI 일정 생성 → 근거 확인 → 조건 수정 → 가격/재고 재검증
5. AI 스토리 초안 → 편집자 검수 → 발행 → 원천 회수 후 재검토

## 7. 격리·보안 테스트

- 모든 파트너 API에 다른 `partnerId/providerId/placeId`를 경로·쿼리·본문으로 주입
- PMS 채널 토큰의 tenant와 요청 tenant 불일치
- 목록/합계/자동완성/내보내기/파일 URL/로그/캐시/검색/RAG의 간접 누출
- IDOR, 권한 상승, 역할 변경 후 기존 세션, API 키 회전/폐기
- CSRF/XSS/SQL·NoSQL injection/SSRF/업로드 파일 검증/속도 제한
- 프롬프트 인젝션, 도구 인자 변조, 시스템 프롬프트/비밀정보 탈취
- 개인정보 마스킹, 보존/삭제, 감사 추적, 로그 변조 방지

운영과 유사한 보안 테스트에는 실제 고객 개인정보/운영 자격증명을 사용하지 않는다.

## 8. 장애·복구 테스트

- PMS/외부 공급자 timeout, 지연, 5xx, 부분 필드, 잘못된 버전
- 캐시 만료, 큐 지연/중복, worker 재시작, DB 일시 장애
- 이벤트 중복/순서 역전/유실 재생
- 예약 성공 응답 유실과 결제 webhook 중복
- 회로 차단기의 열림/반열림/복구와 신규 예약 차단
- 백업 복원, 마이그레이션 실패/롤백, 검색 인덱스 재구축
- 공급 장애 중 기존 예약 상태조회/고객지원의 우선 처리

장애 시 허위 가격·재고·예약확정을 보여주지 않는 것이 가용성보다 우선한다.

## 9. 성능 기준

초기 목표는 실제 트래픽 관측 후 조정하되 릴리스별 부하 모델과 함께 기록한다.

| 대상 | 초기 기준 |
|---|---|
| 공개 캐시 가능 조회 | p95 500ms 이하 |
| 실시간 PMS Offer 채널 | Shadow 기준 p95 750ms 이하 |
| 공개 API 유효 응답률 | 월 99.9% 목표; 공급자 장애는 별도 표시 |
| 검색 | 대표 필터 조합에서 p95 800ms 이하 |
| 예약 명령 | 사용자에게 진행 상태 제공, 동기 대기 상한 후 대사 전환 |
| 비동기 작업 | 지연/적체/실패율 경보와 재처리 가능 |

성능 시험은 평균만 보지 않고 p95/p99, 오류율, DB/캐시/큐 포화, 공급자별 지연을 기록한다.

## 10. 접근성·국제화·호환성

- 키보드만으로 검색/필터/객실 선택/예약/취소 가능
- 스크린리더 이름/역할/상태, 포커스 이동, 오류 연결, 대비, 확대/리플로우
- 한국어/영어를 시작으로 문자열 누락, 긴 번역, 복수형, RTL 확장 가능성
- 호텔 시간대와 사용자 시간대의 날짜 경계, DST, 자정, 통화 반올림
- 주요 데스크톱/모바일 브라우저와 반응형 뷰포트
- 구조화 데이터와 canonical/언어 대체 링크 검증

## 11. AI/RAG 평가

- 고정 골든 세트와 릴리스 후보 비교를 사용한다.
- 검색 Recall@K/nDCG, 근거 포함/정확성, 폐기 원천 노출, 환각률을 측정한다.
- 일정의 시간·이동·운영시간·예산·접근성 제약 통과율을 측정한다.
- 가격/재고 실시간 재검증 누락과 AI 직접 거래 실행은 허용 0건이다.
- 모델/프롬프트/도구/인덱스 버전별 비용·지연·신고·사용자 수정률을 비교한다.

## 12. Hotel_PMS 회귀 게이트

HotelNGo 변경으로 Hotel_PMS를 수정하는 단계에서는 다음을 모두 통과해야 한다.

- 기존 `/api/v1` Mock 계약과 화면 호출 회귀
- 예약 CRUD/상태 전이/날짜 충돌 회귀
- 객실상태·요금·예약의 데이터 일관성
- 새로고침/저장 지속성 및 기존 localStorage 흐름
- 프런트데스크 주요 업무 E2E
- 채널 API가 내부 운영 API/테넌트/권한에 미치는 영향

Release 1에는 PMS 파일을 변경하지 않으므로 변경 없음 검증을 인수 증거로 남긴다.

## 14. CI/CD와 증거

1. 정적 검사와 단위 테스트
2. 스키마/마이그레이션 및 API 호환성 검사
3. 어댑터 소비자 계약 테스트
4. 격리/보안 핵심 통합 테스트
5. 핵심 E2E와 접근성 자동검사
6. staging 회귀·부하·복구·AI 평가
7. 승인된 마이그레이션/기능 플래그로 점진 배포

릴리스 증거에는 커밋/빌드, 요구사항 ID, 테스트 결과, OpenAPI/스키마 버전, 마이그레이션, 알려진 제한, 롤백/비활성화 방법, 관측 대시보드 링크를 포함한다.
