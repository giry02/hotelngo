# AI 여행 구축 검토 및 실행 리포트

## 결론

AI 여행은 구현할 수 있다. 그러나 LLM에 여행 문장을 바로 보내는 것만으로는 예약 가능한 서비스가 되지 않는다. 먼저 **여행 입력 계약, 신뢰할 장소·상품 데이터, 실시간 도구, 제약 검증, 평가와 운영 관제**가 필요하다.

이번 구현은 그 기반을 검증하는 `DETERMINISTIC_MOCK` 단계다. `ai-travel.html`은 실제로 입력을 받고 여행 속도·예산·도시 키워드에 따라 일정을 다시 만들며, 결과에 사용한 기반과 미확인 정보를 표시한다. 실제 LLM이나 RAG를 사용했다고 표현하지 않는다.

## 현재 구현

- 화면: `ai-travel.html`
- 실행 코드: `scripts/ai-travel.js`
- 설정 계약: `data/mock/ai/travel-planner-config.json`
- 인증 경계: 결과 조회는 비회원도 가능, `내 여행에 복사`는 로그인 필요
- 지원 예시 도시: 교토, 다낭, 방콕
- 적용 규칙: 여행 속도별 하루 최대 방문 수, 예산 성향, 도시 템플릿
- 결과 근거: 장소 Mock, 이동시간 추정치, 규칙 버전
- 명시적 제한: 운영시간·날씨·실시간 가격·재고는 미연결

## 실제 AI 전에 필요한 기반

| 기반 | 필요한 이유 | 완료 기준 |
|---|---|---|
| 여행 요청 스키마 | 자유 문장을 기간·인원·취향·금지조건으로 구조화 | JSON Schema와 버전 정책 |
| Place 지식 모델 | 이름만 같은 장소, 폐업, 지점 중복 방지 | canonical ID, 좌표, 카테고리, 출처, 확인일 |
| Product/Offer 연결 | 추천을 실제 상품과 연결 | 상품 ID와 공급 최신성 포함 |
| 운영시간 도구 | 휴무일·마지막 입장 오류 방지 | 출처·확인 시각을 반환하는 Tool API |
| 경로/이동 도구 | 불가능한 이동과 과도한 걷기 방지 | 교통수단별 시간·거리·신뢰도 |
| 날씨 도구 | 우천·폭염·태풍 대체 일정 | 날짜별 예보와 대체 규칙 |
| 호텔 공급 도구 | 허위 가격·재고 추천 방지 | `HotelSupplyAdapter` 조회 계약 |
| RAG 출처 체계 | 설명의 근거·최신성·권리 관리 | 문서 ID·인용·유효기간 |
| 평가 세트 | 보기 좋은 결과와 가능한 결과 구분 | 충돌·환각·선호도·안전 지표 |
| 개인정보/동의 | 위치·동행·아동 정보 안전 처리 | 최소수집·삭제·동의 이력 |
| 관측·비용 통제 | 실패·지연·비용을 운영 | trace, tool 비용, cache, fallback |

## 권장 구현 순서

1. 현재 JSON 입력·출력 계약을 API DTO로 고정한다.
2. Place/Content 카탈로그에 출처·확인일·권리·좌표를 추가한다.
3. 경로·운영시간·날씨를 읽기 전용 Tool API로 연결한다.
4. Hotel_PMS Mock 공급 계약으로 호텔 후보의 요금·재고를 재검증한다.
5. 규칙 엔진을 유지한 채 LLM을 후보 생성과 설명에만 제한 적용한다.
6. 규칙 검증기가 LLM 결과를 거부·수정하고 신뢰 상태를 붙인다.
7. 골든 요청과 공급 장애 평가를 통과한 뒤 일부 사용자에게 공개한다.

## 목표 흐름

```text
사용자 문장
  → Request Parser (기간/인원/취향/제약)
  → Candidate Retriever (Place + Content RAG)
  → Tool Calls (route/hours/weather/hotel supply)
  → Itinerary Composer (LLM 선택 가능)
  → Deterministic Validator (시간·거리·예산·재고)
  → 근거·신뢰·확인시각이 포함된 일정
  → 로그인 후 My Trip 저장
```

LLM은 원천 데이터나 예약 시스템이 아니다. 가격·재고·운영시간을 임의로 생성하지 못하도록 하고, 확정 단계에서는 항상 공급자 API를 다시 조회해야 한다.

## 사용자 경험 원칙

- 자유 문장 + 속도 + 예산으로 시작한다.
- 결과에 추천 이유, 확인한 정보, 미확인 정보를 함께 보여준다.
- 장소 교체는 해당 시간대 제약 안에서 후보를 제시한다.
- 로그인은 결과 생성 전이 아니라 저장·공유·협업 시점에 요구한다.
- 재고 hold와 공급자 확정 전에는 `예약 확정`으로 표시하지 않는다.
