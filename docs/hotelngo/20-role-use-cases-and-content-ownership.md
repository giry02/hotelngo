# 역할별 유즈케이스와 콘텐츠 소유권

## 1. 목적

이 문서는 HotelNGo 화면을 “관리자에게 기능을 몰아넣는 방식”으로 만들지 않기 위한 구현 기준선이다. 각 화면은 실제 사용 주체, 원천 데이터, 승인 주체, 공개 결과가 명확해야 한다.

## 2. 시스템과 계정 경계

| 영역 | 사용자 | 계정 영역 | 주 책임 |
|---|---|---|---|
| HotelNGo B2C | 여행자·회원 | `HOTELNGO_MEMBER` | 검색, 저장, 일정 작성, 공유, 복사, 예약 |
| 호텔 콘텐츠센터 | 호텔 홍보·판매 담당자 | `HOTELNGO_HOTEL` | PMS 데이터를 보완하는 공개 콘텐츠·사진·시설·정책 |
| 비호텔 파트너센터 | 골프장·음식점·차량·스파·투어 운영자 | `HOTELNGO_PARTNER` | 장소, 판매상품, 자원, 운영시간, 슬롯, 재고, 예약 |
| 플랫폼 관리자 | HotelNGo 운영자 | `HOTELNGO_ADMIN` | 입점·소유권·공개·신고·품질 심사와 감사 |
| Hotel_PMS | 호텔 프런트·운영 직원 | PMS 독립 계정 | 객실유형, 요금, 재고, 숙박 예약과 현장 운영 |

PMS 직원 계정, HotelNGo 호텔 콘텐츠 계정, B2C 회원 계정은 서로 재사용하지 않는다.

## 3. 데이터 소유권

### 3.1 PMS 원천 데이터

Hotel_PMS가 원천인 데이터는 HotelNGo에서 임의 수정하지 않는다.

- `Tenant`: 호텔 운영 단위
- `RoomType`: PMS 객실유형 식별자와 운영 명칭
- 날짜별 요금과 가용 수량
- 예약 상태

### 3.2 HotelNGo 호텔 공개 콘텐츠

호텔 콘텐츠센터가 소유하며 PMS 식별자에 덧붙이는 오버레이 데이터다.

- 호텔 대표·공간·부대시설 사진과 영상
- 다국어 호텔 소개, 핵심 특징, 주변 정보
- 공개 편의시설, 체크인 안내, 반려동물·아동·접근성 정책
- `pmsRoomTypeId`별 객실 사진 여러 장
- 객실 크기, 침대, 전망, 욕실, 수용인원, 객실 편의시설 설명
- 사진 순서, 대표 사진, 대체 텍스트, 공개 상태

객실유형 매핑 키는 `tenantId + pmsRoomTypeId`다. PMS 원천의 요금·재고와 HotelNGo 공개 콘텐츠는 조회 시 합성하며 원본을 복제하거나 덮어쓰지 않는다.

### 3.3 비호텔 파트너 데이터

비호텔 파트너는 다음 계층을 직접 관리한다.

`Provider → Place → Product → Resource → Schedule → Offer → Inventory → Booking`

- 골프: 코스·티박스·티타임·팀 수용량
- 음식점: 지점·테이블/룸·운영시간·예약 슬롯
- 차량: 차량·기사·서비스 지역·배차 가능시간
- 스파: 룸·테라피스트·프로그램·예약 슬롯
- 투어: 일정·가이드·언어·집결지·출발 회차

호텔은 이 파트너 계층의 업종 메뉴에서 제외한다. PMS 연계 호텔과 PMS 없는 호텔 모두 호텔 콘텐츠센터에서 공개 콘텐츠를 관리하고, 공급 방식만 `INTERNAL_PMS` 또는 `PARTNER_MANAGED`로 구분한다.

### 3.4 회원 작성 콘텐츠

회원이 만든 일정·후기·스토리의 작성자는 회원이다. 플랫폼 관리자는 대신 작성하는 주체가 아니라 공개 심사·신고·품질 관리 주체다.

## 4. 핵심 유즈케이스

### UC-HOTEL-01 호텔 상세 탐색

1. 여행자가 호텔을 검색한다.
2. 호텔 전체 사진을 공간별로 본다.
3. 객실유형마다 여러 사진과 크기·침대·전망·욕실·정원을 확인한다.
4. 날짜별 PMS 요금·가용수량과 HotelNGo 공개 콘텐츠를 함께 본다.
5. 원하는 객실을 선택해 예약 흐름으로 이동한다.

인수 조건:

- “사진 모두 보기”는 실제 갤러리를 열고 표시된 장수와 렌더링 장수가 일치한다.
- 각 객실유형에 독립된 사진 배열이 있고 키보드와 모바일에서도 열 수 있다.
- PMS 미응답·오래된 데이터·매진은 명시적으로 표시한다.

### UC-HOTEL-02 PMS 연계 호텔 콘텐츠 보완

1. 호텔 담당자가 HotelNGo 호텔 계정으로 로그인한다.
2. 자기 호텔의 PMS 연결과 객실유형 매핑을 조회한다.
3. 호텔 공개 소개·사진·시설·정책을 작성한다.
4. PMS 객실유형별로 여러 사진과 공개 설명을 등록한다.
5. 미리보기 후 공개 심사를 요청한다.
6. 관리자가 소유권과 품질을 확인해 공개한다.

인수 조건:

- 호텔 담당자는 PMS 요금·재고를 수정할 수 없다.
- 사진은 대표 지정, 순서 변경, 삭제, 대체 텍스트를 지원한다.
- 객실유형 매핑이 끊긴 콘텐츠는 공개되지 않고 “매핑 필요”로 남는다.

### UC-PARTNER-01 비호텔 업체 입점

1. 업체 담당자가 업종을 선택해 별도 파트너 계정을 만든다.
2. 사업자·정산·소유권 증빙을 제출한다.
3. 관리자가 승인한다.
4. 승인된 업체 범위에서만 장소·상품·자원·일정을 등록한다.

### UC-PARTNER-02 예약 가능 슬롯 구성

1. 업체가 예약을 수행하는 실제 자원(테이블, 차량, 가이드, 룸, 코스)을 등록한다.
2. 자원별 운영시간과 휴무·예외일을 정한다.
3. 상품과 자원을 연결하고 회차·슬롯·수용량·가격을 만든다.
4. B2C와 AI 일정은 공개된 슬롯만 예약 가능으로 판단한다.

### UC-PARTNER-03 업종별 선택 상품 구성

파트너나 초기 운영 관리자는 장소 소개만 등록하는 것이 아니라 여행자가 비교·선택할 수 있는 판매 단위를 등록한다.

| 업종 | 여행자 선택 단위 | 필수 상품 정보 | 실제 자원·슬롯 |
|---|---|---|---|
| 골프 | 코스, 9/18/27홀, 날짜, 티타임, 인원, 옵션 | 난이도, Par, 그린피, 캐디·카트 포함 여부, 클럽 대여, 픽업, 복장, 취소 | 코스·티박스·팀 수용량·티타임 |
| 차량 | 차종, 대여/기사 포함 방식, 픽업·반납, 기간 | 좌석·수하물, 보험, 주행거리, 기사, 공항/지역 할증, 보증금 | 차량·기사·서비스 지역·배차시간 |
| 음식점 | 지점, 메뉴/코스, 날짜, 시간, 좌석/룸 | 요리 유형, 메뉴 사진, 구성, 알레르기, 인원, 최소 주문, 결제 방식 | 테이블·룸·회차별 좌석 |
| 마사지·스파 | 프로그램, 시간, 인원, 룸, 픽업 | 기법, 소요시간, 포함사항, 임산부·연령 제한, 테라피스트 요청 | 싱글/커플룸·테라피스트·시간 슬롯 |
| 투어·체험 | 코스, 출발 회차, 언어, 인원, 옵션 | 일정, 소요시간, 집결/픽업, 가이드 언어, 포함·불포함, 연령 제한 | 가이드·차량·회차 수용량 |

가격 방식은 다음 중 하나를 반드시 가진다.

- `FIXED_PRICE`: 즉시 확인 가능한 고정가
- `FROM_PRICE`: 옵션에 따라 달라지는 시작가
- `REQUEST_QUOTE`: 요청 후 견적
- `PAY_ON_SITE`: 현장 결제
- `TO_BE_CONFIRMED`: 운영자가 추후 확정

“가격 없음”으로 표시하지 않고 판매 방식과 확정 시점을 설명한다.

### UC-PARTNER-04 호텔 결합 혜택

1. 파트너가 적용 가능한 호텔 또는 지역, 숙박일 조건을 선택한다.
2. 할인 유형을 정액·정률·무료 옵션·픽업 포함 중 선택한다.
3. 호텔 예약이나 일정 조건이 충족되면 B2C에 결합 혜택을 표시한다.
4. 카트와 주문에는 원가, 할인, 부담 주체, 정산 기준을 별도 기록한다.

호텔과 파트너 상품을 같은 재고로 합치지 않는다. 각각 재검증·예약한 뒤 하나의 여행 주문에서 연결한다.

### UC-PARTNER-05 예약과 정산

1. 고객이 상품 옵션과 슬롯을 선택한다.
2. 공급자의 판매 방식에 따라 즉시예약, 요청예약, 견적요청으로 분기한다.
3. 예약에는 `providerId`, `placeId`, `productId`, `resourceId`, `scheduleId`, `offerId`를 저장한다.
4. 확정·이용완료·취소 상태가 정산 대상 여부를 결정한다.
5. 정산에는 판매가, 쿠폰, 결합 할인, 수수료, 세금, 파트너 지급액을 분리한다.

초기 운영자가 대신 등록한 `PLATFORM_CURATED` 장소도 같은 모델을 사용한다. 업체가 입점·소유권 인증을 완료하면 승인된 범위만 파트너 계정으로 이관한다.

### UC-TRIP-01 회원 일정 작성

1. 로그인한 회원이 날짜·도시·동행자를 정해 여행을 만든다.
2. 호텔·음식점·투어·장소를 날짜와 시간에 배치한다.
3. 이동시간, 영업시간, 예약 가능 여부를 점검한다.
4. 개인 메모를 포함한 초안을 저장한다.

### UC-TRIP-02 일정 공개와 공유

1. 작성자가 공개 제목·커버·설명·태그를 입력한다.
2. 개인 메모와 예약번호 등 비공개 필드를 제외한다.
3. `링크 공유` 또는 `공개 발행`을 선택한다.
4. 공개 발행은 기본 품질 검사를 거쳐 커뮤니티에 노출된다.

상태:

`DRAFT → LINK_SHARED | PENDING_REVIEW → PUBLISHED → ARCHIVED`

### UC-TRIP-03 다른 회원 일정 복사

1. 여행자가 공개 일정을 탐색한다.
2. 일정 상세에서 날짜별 코스와 예약 가능 여부를 본다.
3. “내 여행으로 복사”를 누른다.
4. 로그인 후 원본과 독립된 새 `DRAFT`가 만들어진다.
5. 출발일·인원 변경 시 가격·재고·운영시간을 다시 검증한다.

원본 작성자, 출처 일정 ID, 복사 시각은 기록하되 이후 편집은 원본을 변경하지 않는다.

### UC-TRIP-04 일정 예약 준비

1. 일정의 각 항목을 `즉시예약`, `요청예약`, `정보형`, `확인 필요`로 분류한다.
2. 날짜·인원 기준으로 공급자 재고와 영업 여부를 재검증한다.
3. 예약 가능한 항목만 여행 카트에 추가한다.
4. 정보형 장소는 자동 예약 성공으로 표시하지 않는다.
5. 실제 다건 예약은 공급자별 hold·결제·실패 보상 규칙이 준비된 뒤 활성화한다.

### UC-ADMIN-01 공개 품질과 신고 검수

관리자는 호텔 콘텐츠, 파트너 상품, 회원 공개 일정을 별도 큐에서 검토한다. 작성자가 아니며 다음만 수행한다.

- 소유권·금지 콘텐츠·중복·기본 품질 검수
- 승인·수정 요청·비공개·신고 처리
- 근거와 처리자를 포함한 감사 로그 기록

## 5. AI 여행 입력과 출력 경계

AI 일정은 다음을 입력으로 사용한다.

- 검증된 호텔 공개 콘텐츠
- 실시간 공급 어댑터가 반환한 요금·재고·슬롯
- 회원이 공개를 허용한 일정
- 운영시간·이동시간·휴무·연령 제한

AI는 편집 가능한 일정 초안을 만들 수 있지만 가격·재고·영업 여부를 창작하지 않는다. 자동 예약 전에 각 공급자 어댑터로 재검증하며, 예약할 수 없는 항목은 정보형으로 남긴다.

## 6. 이번 프로토타입 구현 범위

- 호텔 전체 및 객실유형별 다중 이미지 갤러리
- PMS 매핑을 읽기 전용으로 보여주는 호텔 콘텐츠센터
- 호텔·객실 공개 콘텐츠와 사진 편집 Mock
- 비호텔 파트너의 자원·운영시간·슬롯 화면
- 회원 공개 일정 목록·상세·복사·발행·예약 준비 화면
- 회원 일정 공개 심사 관리자 화면

외부 이미지 업로드, 실제 PMS 쓰기, 실제 PG 결제, 공급자 실예약, LLM 호출은 Mock 경계로 유지한다.
